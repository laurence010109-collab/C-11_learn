整体
  这个项目本质上是一个“C++ 后端编排器 + Linux 系统工具链”的工程系统。C++ 负责 HTTP API、配置解析、状态机、并发调度和文件落盘；Linux 负责进程、信号、文件系统、环境变量和多媒体工具运行。核
  心入口在 cpp-server/src/main.cpp:1，构建配置在 cpp-server/CMakeLists.txt:1。

  C++ 技术

  - C++17：项目明确使用 C++17 标准，很多实现建立在现代 STL 之上，比如 filesystem、optional、thread、future、condition_variable。见 cpp-server/CMakeLists.txt:4。
  - STL 容器与字符串处理：大量使用 std::vector、std::map、std::unordered_map、std::queue、std::string、std::ostringstream 来组织运行态、请求体、事件流和命令参数。它们是整个后端数据结构的
    基础。见 cpp-server/src/main.cpp:10。

  - std::filesystem：负责路径拼装、目录创建、文件是否存在判断、原子重命名、日志目录和数据目录管理。这个项目几乎所有 cases/、runs/、config/、frames/ 的落盘都依赖它。见 cpp-server/src/
    main.cpp:619, cpp-server/src/vlm_data_paths.cpp:26。

  - 文件 I/O：用 std::ifstream / std::ofstream 读写 YAML、JSON、prompt、图片和事件文件。项目采用“临时文件 + rename”的原子写法，避免状态文件写到一半被其他线程或进程读到。见 cpp-server/
    src/main.cpp:604, cpp-server/src/main.cpp:619。

  - JSON 库 nlohmann/json：后端 REST API 的请求解析、响应组装、events.jsonl、manifest.json、数据集清单等，基本都靠这个库完成。见 cpp-server/src/main.cpp:7, cpp-server/src/
    vlm_data_paths.cpp:6。

  - YAML 库 yaml-cpp：用来读取 aibox.yaml、scenario_profiles.yaml、case.yaml，把“场景模板 + Case 覆写 + 运行时参数”合并成最终执行配置。见 cpp-server/src/main.cpp:8, cpp-server/src/
    main.cpp:479。

  - HTTP 服务端 cpp-httplib::Server：项目没有自己手写 socket 服务器，而是用 cpp-httplib 暴露 /api/health、/api/runs、/api/cases、/api/config/* 等接口。它是整个后端 API 层的核心。见 cpp-
    server/src/main.cpp:5630。

  - HTTP 客户端 cpp-httplib::Client / SSLClient：用于请求上游 OpenAI 兼容模型网关，发 Chat Completions 请求、做连通性测试、控制超时。见 cpp-server/src/main.cpp:400, cpp-server/src/
    main.cpp:6027。

  - OpenSSL：一部分用于给 cpp-httplib 提供 HTTPS/TLS 能力，另一部分直接被代码拿来做 base64 和 AES-256-CBC 解密。项目支持对 prompt 文件加密存储，运行时自动解密。见 cpp-server/
    CMakeLists.txt:23, cpp-server/src/main.cpp:248, cpp-server/src/main.cpp:566。

  - 并发编程：项目大量使用 std::thread、std::mutex、std::condition_variable、std::future、std::promise。最典型的是 LaneExecutor，它实现了单线程顺序执行队列，把落盘和推理从 HTTP 工作线程
    里拆出去。见 cpp-server/src/main.cpp:4572。

  - 原子/锁：sig_atomic_t 用于停服信号，std::mutex 用于 YAML 解析、日志写入、run 状态、Sentry 流状态等共享数据保护。项目是典型的多线程共享状态后端。见 cpp-server/src/main.cpp:71, cpp-
    server/src/vlm_log.cpp:17。

  - chrono 时间体系：用于超时、耗时统计、日志时间、自动推理节拍、事件时间戳和文件时间展示。项目里 http_ms、assemble_ms、schedule_duration_ms 这些性能字段都基于它。见 cpp-server/src/
    main.cpp:497, cpp-server/src/vlm_log.cpp:26。

  - 异常处理：后端给 httplib 配了统一异常处理器，避免单个 handler 崩掉整个服务，同时把错误写日志并返回 500。见 cpp-server/src/main.cpp:5631。
  - 日志系统：项目有自己的一套 C++ 日志子系统，支持按天滚动、最小日志级别、HTTP 请求日志、控制台镜像输出。见 cpp-server/src/vlm_log.cpp:146。
  - CMake 工程化：后端不是裸 g++ 编译，而是用 CMake 管依赖、编译选项、PCH、ccache、线程库和 OpenSSL。见 cpp-server/CMakeLists.txt:11, cpp-server/CMakeLists.txt:38, cpp-server/
    CMakeLists.txt:111。

  - FetchContent：三方依赖 cpp-httplib、nlohmann/json、yaml-cpp 通过 CMake 的 FetchContent 拉取，并支持离线 tarball 缓存。作用是降低外部依赖管理复杂度。见 cpp-server/CMakeLists.txt:41,
    cpp-server/CMakeLists.txt:105。

  Linux 技术

  - POSIX 进程模型：项目显式用 fork + execvp + waitpid 启动外部命令，而不是简单 system()。这样更可控，能拿到退出码，也能决定是否捕获输出。主要用于调 ffmpeg、ffprobe 和 shell 命令。见
    cpp-server/src/main.cpp:114, cpp-server/src/main.cpp:207。

  - pipe + dup2：子进程的 stdout/stderr 被重定向到管道，父进程再读取输出。这套机制用来抓取 ffmpeg/ffprobe 的结果与错误信息。见 cpp-server/src/main.cpp:119, cpp-server/src/main.cpp:132。
  - epoll + 非阻塞 I/O：这里不是拿 epoll 做 HTTP 服务器，而是拿它监听子进程输出管道，避免读外部命令输出时卡死。算是一个典型 Linux 系统编程点。见 cpp-server/src/main.cpp:150。
  - fcntl(O_NONBLOCK)：把管道读端设成非阻塞，配合 epoll_wait 安全读取。见 cpp-server/src/main.cpp:154。
  - 信号处理 sigaction：处理 SIGINT、SIGTERM 做优雅停服，处理 SIGSEGV、SIGABRT 等致命信号做崩溃诊断，同时忽略 SIGPIPE 防止断连导致进程退出。见 cpp-server/src/main.cpp:84, cpp-server/src/
    main.cpp:459。

  - async-signal-safe 写终端：信号处理函数里用 write(STDOUT_FILENO, ...)，这是标准 Linux/POSIX 思路，避免在 signal handler 中做不安全操作。见 cpp-server/src/main.cpp:74。
  - syslog：除了自己的文件日志，项目还接入了 Linux syslog，方便服务启动/停止事件进入系统日志。见 cpp-server/src/main.cpp:95。
  - 环境变量驱动运行时：大量用 getenv/unsetenv 控制运行，例如 VLM_DATA_DIR、VLM_API_KEY、VLM_KEEP_PROXY、VLM_LOG_*。这是很典型的 Linux 服务配置方式。见 cpp-server/src/main.cpp:472, cpp-
    server/src/main.cpp:5579, cpp-server/src/vlm_log.cpp:148。
    vlm_data_paths.cpp:26。

  - Bash 运维脚本：项目在 Linux 上大量依赖 shell 脚本做开发、启动、构建、发布、清理和依赖安装。不是简单辅助，而是整个工程工作流的一部分。见 scripts/dev.sh:1, scripts/build.sh:1, scripts/
    start.sh:1。

  - flock 文件锁：dev.sh 用它防止你重复启动多个后端进程，避免同端口多实例导致 run 状态错乱。见 scripts/dev.sh:21。
  - apt-get / build-essential / g++ / cmake / pkg-config / libssl-dev / ffmpeg：这套是项目在 Debian/Ubuntu 上的标准依赖安装方案。也就是说，这个项目明显是按 Linux 开发环境设计的。见
    scripts/install-deps.sh:24。

  - chmod +x 和可执行位修复：构建脚本会修复某些环境下 vite` 执行位丢失的问题，这也是 Linux 文件权限模型的直接体现。见 scripts/build.sh:84。
  - WSL/Linux 兼容处理：脚本里专门检测 /proc/version、DNS、代理和本地 ffmpeg 路径，说明作者明确把 Linux/WSL 作为主要运行平台。见 scripts/dev.sh:43。
  - 本地静态/媒体文件服务：后端直接从 Linux 文件系统读取图片、视频、缩略图、事件文件并经 HTTP 返回，而不是接对象存储或数据库。见 cpp-server/src/main.cpp:554, cpp-server/src/
    main.cpp:1135, cpp-server/src/main.cpp:8440。

  - JSONL 追加日志：events.jsonl 用追加写方式记录每次推理事件，这是一种很典型的 Linux 文件化事件流实现，简单、可 grep、可回放。见 cpp-server/src/main.cpp:542, cpp-server/src/
    main.cpp:4495。

  最关键的技术组合

  - C++ 负责“业务编排层”：API、配置、状态机、线程、日志、文件落盘。
  - Linux 负责“执行环境层”：进程、信号、文件系统、环境变量、锁、命令行工具。
  - ffmpeg/ffprobe 负责“重活”：抽帧、视频探测、图像预处理。
  - cpp-httplib + OpenSSL 负责“网络层”：本地 REST 服务和上游模型 HTTPS 调用。
